# AI Learnings — IQ Option Bot

### [2026-06-16] Wrong win detection function for turbo options
- **Context:** Used `check_win_digital_v2(tid)` for 1-min turbo option trades. All trades showed as losses.
- **Why It Failed:** `check_win_digital_v2` is for DIGITAL options only. For turbo options created by `api.buy(amount, asset, direction, 1)`, it returns `(False, None)` every time — never checks the real result.
- **The Heuristic:** Use `check_win_v4(tid)` for turbo options (`api.buy()`), use `check_win_digital_v2(tid)` only for digital options (`api.buy_digital_spot()`).

### [2026-06-16] While-true loop with new API connection every cycle causes server throttling
- **Context:** rho_real.py created a new `IQ_Option()` connection every 15 seconds in a while-true loop. After hours of this, IQ Option blocked the IP.
- **Why It Failed:** Creating and destroying websocket connections every 15-35 seconds exhausts the server's tolerance. The server silently blocks the IP after sustained aggressive reconnection.
- **The Heuristic:** Use single-run pattern (one connection, one trade, exit) with a bash loop and 45-60s sleep between runs. Match the zeta3_real.py pattern.

### [2026-06-16] iqoptionapi has infinite loops that hang forever on connection failure
- **Context:** When IQ Option's websocket stops responding, `api.connect()` hangs forever at `while global_value.balance_id == None: pass` and `while self.timesync.server_timestamp != None: pass`.
- **Why It Failed:** The iqoptionapi library has 5+ infinite while-true loops with no timeout in connect(), start_websocket(), and send_ssid(). If the server doesn't respond, the bot hangs permanently.
- **The Heuristic:** Always run iqoptionapi bots with `timeout 180 python3 script.py` in the bash wrapper. Never run without a timeout guard.

### [2026-06-16] IPv6 DNS can resolve to blocked addresses while other addresses work
- **Context:** iqoption.com has 3 IPv6 addresses. DNS resolved to `2a00:1098:*` which was blocked, while `2a01:4f8:c2c:123f:*` worked fine. HTTP worked (curl retries different IPs via happy eyeballs) but websocket hung (tries first IP only).
- **Why It Failed:** The websocket library resolves DNS once and uses that single IP. If it picks a blocked address, all connections fail. curl worked because it tries multiple resolved addresses.
- **The Heuristic:** When IPv6 connection fails but curl works, check `/etc/hosts` for DNS override with the working address. Test with `curl -6 -v` to see which IP actually connects.

### [2026-06-16] CALL signals underperform PUT signals in 1-min S/R bounce strategy
- **Context:** rho_bounce strategy tested both CALL (support bounce) and PUT (resistance rejection). CALL was 2W/5L=29%, PUT was 75%.
- **Why It Failed:** Support bounces in 1-min timeframe are weaker than resistance rejections. Sellers push back harder at resistance than buyers push back at support. Market microstructure favors continuation after rejection.
- **The Heuristic:** In S/R bounce strategies for short-expiry binary options, disable CALL (support bounce) unless proven otherwise with 20+ trade sample.

### [2026-06-16] Practice account results don't predict real account results
- **Context:** Momentum 5-min strategy was 78% on practice but 33% on real. The practice version picked crypto assets (INJUSD, DOTUSD, TRON) while real picked stocks (GOOGLE, JPM).
- **Why It Failed:** Practice and real accounts scan independently at different times, hitting different assets. Crypto assets in practice had stronger momentum patterns than stocks in real.
- **The Heuristic:** Never trust practice win rate as a predictor of real performance. Test on real (small amount) to validate. Different asset pools = different results.

### [2026-06-16] Night market (00:00-07:00 UTC) kills ranging strategies
- **Context:** rho_bounce was 78% during active hours (12:00-18:00 UTC) but 33% at night (03:00 UTC). Lost 4 trades in a row.
- **Why It Failed:** Low volume at night means less predictable price action. S/R levels from active hours become unreliable. ADX readings are misleading in thin markets.
- **The Heuristic:** Add time filter to S/R strategies: only trade 07:00-18:00 UTC. Or at minimum, reduce position size outside active hours.
